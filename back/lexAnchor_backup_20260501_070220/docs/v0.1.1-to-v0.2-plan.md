# LexAnchor v0.1.1 to v0.2 Plan

## 1. Version Principle

LexAnchor uses two kinds of version movement:

```text
Anchor maturity versions: v0.1 -> v0.2 -> v0.3 -> v1.0
Engineering completion versions: v0.1.1, v0.2.1, v0.3.1
```

`v0.1.1` does not add a new anchor type. It completes the delivery loop for the v0.1 anchor product.

## 2. Corrected Anchor Roadmap

| Version | Anchor Scope | Meaning |
| --- | --- | --- |
| v0.1 | Text Anchor, Missing Anchor, Risk Anchor, Context Anchor | Minimal product-grade review structure. |
| v0.1.1 | No new anchor type | Engineering completion for v0.1 artifacts and E2E reliability. |
| v0.2 | Semantic Anchor, Escalation Anchor | Start model-assisted interpretation and explicit review routing. |
| v0.3 | Obligation Anchor, Relation Anchor | Extract duties and cross-clause dependencies. |
| v1.0 | Action Anchor | Productize decisions into workflow actions. |

## 3. Current v0.1 State

Implemented server-side capabilities:

- FastAPI HTTP service.
- HTTP API documentation.
- Rule loading with base and industry overlays.
- Text, Missing, Risk, and lightweight Context anchors.
- Semantic candidate output without confirmed semantic judgment.
- Docling integration point for document extraction.
- LangExtract integration point for evidence grounding.
- JSON and Markdown report artifacts.
- SQLite job store.
- Text redaction preview.

Known v0.1 product gap:

- The server does not yet generate annotated PDF artifacts from PDF scans.
- PDF layout grounding and rendering still need migration from the demo POC into server artifact services.

## 4. v0.1.1 Goal

v0.1.1 completes the v0.1 product loop:

```text
Upload document
  -> Extract text and layout
  -> Generate Text/Missing/Risk/Context anchors
  -> Ground span evidence
  -> Produce JSON report
  -> Produce Markdown report
  -> Produce annotated PDF when source is PDF
  -> Expose artifact URLs over HTTP
```

## 5. v0.1.1 Implementation Tasks

### 5.1 Layout and PDF Artifact Pipeline

- Move or adapt demo `layout_index.py` into `server/app/services`.
- Move or adapt demo `renderer.py` into `server/app/services`.
- Create an `ArtifactService` or extend `ReportService` to own generated files.
- Persist artifact metadata in job result.
- Add `GET /api/v1/jobs/{job_id}/annotated.pdf`.

### 5.2 Document Extraction Hardening

- Make `DocumentService` return text plus layout metadata for PDFs.
- Preserve Docling metadata separately from PDF layout metadata.
- Ensure native PyMuPDF fallback remains available.
- Keep OCR and heavy Docling behavior optional and configurable.

### 5.3 Grounding Hardening

- Keep LangExtract as the preferred network grounding path when enabled.
- Keep local exact-string grounding as deterministic fallback.
- Store grounding metadata on each evidence object.
- Keep document-level Missing Anchors location-free.

### 5.4 Reports and API Docs

- Add annotated PDF URL to scan result artifacts when available.
- Add annotated PDF endpoint to hosted API docs.
- Include anchor-type counts in Markdown report.
- Explicitly list v0.1 anchor types in API docs.

### 5.5 Tests

- Unit test renderer input mapping.
- Unit test artifact endpoint path resolution.
- E2E test text scan.
- E2E test TXT upload scan.
- E2E test PDF upload scan if a stable fixture is available.

## 6. v0.1.1 Definition of Done

v0.1.1 is done when:

1. A PDF upload scan returns `job_id`, JSON report URL, Markdown report URL, and annotated PDF URL.
2. Text anchors render on the original PDF when a layout map is available.
3. Missing anchors appear in a document-level summary page or summary section.
4. Context anchors appear in report summary, not as unsupported legal conclusions.
5. Risk anchors are explicit in every confirmed finding.
6. All artifact endpoints are documented and smoke-tested.
7. Test suite passes with `uv run --python 3.11 python -m unittest`.

## 7. v0.2 Goal

v0.2 starts a new anchor maturity layer. It adds Semantic Anchor and Escalation Anchor.

```text
Semantic Candidate
  -> LLM validation
  -> Confirmed Semantic Anchor or dismissed candidate
  -> Escalation policy
  -> Human review route
```

## 8. v0.2 Implementation Tasks

### 8.1 Semantic Anchor

- Add `SemanticValidatorService` to the server.
- Reuse or adapt the demo LLM validator.
- Store validation metadata:
  - model route
  - model id
  - prompt version
  - confidence
  - evidence spans
  - validation status
- Convert confirmed semantic candidates into `anchor_type: semantic`.
- Keep unconfirmed candidates out of summary score.

### 8.2 Escalation Anchor

- Add explicit escalation policy rules.
- Support escalation reasons:
  - critical severity
  - high severity with human review required
  - semantic uncertainty
  - missing critical clause
  - organization policy override
- Add `escalation_anchor` output section.
- Add review queue metadata without yet building the full team workflow.

### 8.3 API Additions

Potential endpoints:

```text
POST /api/v1/jobs/{job_id}/validate-semantics
GET  /api/v1/jobs/{job_id}/escalations
```

Or controlled scan parameters:

```text
semantic_validation=disabled|candidate|validate
escalation_policy=default|strict|disabled
```

### 8.4 v0.2 Definition of Done

v0.2 is done when:

1. Semantic candidate rules can be validated into confirmed Semantic Anchors.
2. Semantic validation is evidence-grounded and auditable.
3. Escalation Anchors are explicit records, not only fields inside risk objects.
4. Scan reports distinguish confirmed findings, semantic candidates, dismissed candidates, and escalations.
5. Existing v0.1 and v0.1.1 behavior remains backward compatible.

## 9. Recommended Next Step

Start with v0.1.1 task group 5.1:

```text
layout_index migration -> renderer migration -> annotated PDF artifact endpoint
```

This gives the fastest path from a working API server to a reviewable legal product loop.
