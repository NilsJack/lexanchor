# LexAnchor 前端快速上手：给法律背景负责人的产品说明

这份文档写给即将负责 LexAnchor 前端的人。你可能更熟悉合同审查、律所管理或法务流程，而不是后端接口。读它的目的，不是学会写代码，而是理解锚点后台为什么这样设计，以及怎样把这些能力做成律师愿意日常使用的工作台。

阅读目标：

- 理解什么是锚点，为什么它不是普通高亮、不是合同摘要、也不是系统替律师下判断。
- 理解锚点如何帮助律师日常审查合同、管理风险、沉淀规则、推进动作。
- 理解为什么律所、企业法务和律师团队会选择锚点。
- 理解现有后台已经提供哪些能力，前端可以围绕哪些页面和流程来构建。

相关技术文档：

- 完整 API 说明：[api-reference.md](api-reference.md)
- 后台设计说明：[anchor-server-design.md](anchor-server-design.md)

本文概要：

- LexAnchor 的核心不是替律师下结论，而是把法律判断固定在可复核的坐标上。
- 锚点不是 skill，也不是一排工具按钮；它是证据、规则、状态和后续动作的组织方式。
- 律师真正需要的不是更多自动总结，而是少漏点、能复核、能协作、能沉淀团队经验。
- 前端的主形态应是法律审查工作台：文档视图、Anchor 面板、Action Queue、规则治理和审计时间线。
- 未来的产品优势来自规则库和审计链：把资深律师经验变成可运行、可复用、可管理的团队资产。

## 1. 一句话理解 LexAnchor

LexAnchor 的核心不是“让系统读合同后给结论”，而是把律师的审查判断固定在合同中可复核的位置上。

更直白地说：

```text
合同文字 -> 找到关键点 -> 解释法律意义 -> 标出风险/缺失/义务/关系 -> 交给律师判断 -> 形成后续动作
```

锚点的价值在于：它让每一个风险、义务、缺失、升级事项、后续动作都有清楚的来源、理由、位置和状态。

很多工具会直接生成一段总结，例如“这个合同风险较高”。LexAnchor 更关注结论背后的证据链。它会告诉律师：

- 哪一段文字触发了风险。
- 哪个条款缺失导致需要注意。
- 该问题属于什么 anchor 类型。
- 风险等级和置信度是什么。
- 是否需要升级给资深律师或合伙人。
- 它适合转成什么后续 action。
- 谁查看过、谁更新过、什么时候更新过。

所以，锚点前端的第一原则是：不要把后台结果包装成最终意见，而要把它呈现成律师可以复核、可以取舍、可以继续处理的审查证据。

## 2. 什么是锚点

可以把锚点理解成法律工作中的“审查坐标”。

律师审合同时，不只是看文字，还在不断做这些动作：

- 这句话有没有问题？
- 这里缺了什么条款？
- 这个义务是谁承担？期限是什么？
- 这个条款和另一个条款是否冲突？
- 这个风险是否需要升级？
- 这个问题是否需要变成修改意见、谈判点或内部审批事项？

锚点就是把这些动作结构化。

当前 LexAnchor 后台支持这些 anchor 能力：

| Anchor 类型 | 给法律人的解释 | 界面呈现方式 |
| --- | --- | --- |
| Text Anchor | 文字命中点，例如“unlimited liability” | 在合同正文中定位并高亮，同时在侧栏说明规则来源 |
| Missing Anchor | 应有但缺失的条款，例如没有责任上限 | 不要强行绑定某段文字，应显示为“文档级缺失问题” |
| Risk Anchor | 风险判断，例如责任、赔偿、数据、续约风险 | 用严重程度、风险类型、建议动作展示 |
| Context Anchor | 合同语境，例如行业、合同类型、适用场景 | 放在文档概览或顶部摘要，帮助律师理解规则为什么适用 |
| Semantic Anchor | 语义层面的风险，不完全依赖关键词 | 标注为需复核，突出触发理由和置信度 |
| Escalation Anchor | 需要升级审核的事项 | 放入“需升级”列表，显示升级原因和建议角色 |
| Obligation Anchor | 义务，例如返还数据、通知、付款、保密 | 展示义务主体、动作、期限、条件 |
| Relation Anchor | 条款之间的关系，例如终止触发数据返还 | 显示“从哪个条款指向哪个义务/后果” |
| Action Anchor | 可以执行的后续事项 | 进入 action queue，可接受、进行中、完成或关闭 |

设计时不必把所有 anchor 混在一个列表里。律师看到的是不同类型的工作：风险、缺失、义务、升级、动作。界面最好按工作方式组织，而不是按后台字段堆叠。

## 3. 锚点不是 Skill

这里有一个边界要先说清楚：锚点不是 skill，也不适合被做成一组技能按钮。

Skill 是一种能力或操作方法。例如：摘要合同、提取条款、翻译文本、生成修改建议、检查敏感信息。这些都是系统可以执行的动作。

锚点不是动作本身。锚点是法律判断被固定下来的结构化坐标。它回答的不是“系统会做什么”，而是：

- 这个法律问题来自哪里？
- 它依据哪段文字，或者缺少哪个应有条款？
- 它属于什么法律意义：风险、义务、缺失、关系、升级还是行动？
- 它为什么值得律师注意？
- 它后续应该进入什么审查或工作流？
- 它是否已经被律师确认、处理或关闭？

可以这样理解二者关系：

```text
Skill 是工具动作：帮我做一件事。
Anchor 是审查证据：告诉我这件事为什么重要、在哪里、下一步怎么办。
```

例如“提取合同中的责任条款”是 skill；“第 12 条出现无限责任表述，触发责任风险规则，需要律师复核是否加入责任上限”是 anchor。

“生成合同摘要”是 skill；“本合同缺少数据返还义务，因此形成一个 Missing Anchor，并建议进入 action queue”是 anchor。

这一区分会直接影响前端设计。如果把锚点当 skill，页面很容易变成一排按钮：摘要、翻译、检查、生成意见。用户看到的是工具箱，而不是法律工作台。

按锚点来设计，页面会围绕证据、判断、状态和动作来组织：左边是合同，右边是风险、缺失、义务、关系和 action，底部或侧边是审计记录。律师看到的不是“系统会什么”，而是“我的审查工作推进到哪里了”。

所以，LexAnchor 可以调用 skill，但产品核心不是 skill。Skill 是手段，anchor 是法律工作成果的组织方式。

## 4. 一个成熟律师在日常工作中真正想什么

理解锚点前端，可以先看一个资深律师每天处理合同时的真实思考方式。

成熟律师打开一份合同时，通常不是从“这份合同好不好”这个抽象问题开始。他会先在脑中快速建立几个坐标：这是谁的合同、客户站在哪一边、交易金额多大、对方强势程度如何、客户真正不能接受的底线是什么、哪些问题是商业可以接受但法律需要提示的、哪些问题必须升级。

他看条款时，也不是逐字平均用力。他会不断在脑中问：

- 这句话是否改变了风险分配？
- 有没有把本应由对方承担的责任转移给客户？
- 有没有缺少一个成熟交易中通常应有的保护条款？
- 这个义务是否有主体、期限、触发条件和后果？
- 这个条款是否和前后其他条款冲突？
- 这个问题是我能当场处理，还是必须让合伙人、隐私律师、税务律师或业务负责人确认？
- 如果客户以后问我为什么这样改，我能不能拿出清楚证据？

这就是成熟律师和初级审查者的差别：成熟律师不是只找“问题句子”，而是在构建一条可解释的判断链。

```text
看到文字 -> 识别法律意义 -> 判断风险归属 -> 联系客户目标 -> 决定是否升级 -> 形成修改或谈判动作 -> 留下可复核记录
```

锚点要做的，是在这条判断链上减轻律师负担，而不是替代这条判断链。

### 4.1 帮律师少漏点

成熟律师最大的压力之一不是“不会判断”，而是“在大量文件和紧时间下漏掉关键点”。锚点可以把常见高风险条款、缺失条款、义务结构和条款关系先找出来，让律师把注意力放在确认和取舍上。

对应到前端体验，就是让律师一打开扫描结果，就能看到最需要注意的风险、缺失、义务和升级事项，而不是在长合同里重新从零找。

### 4.2 帮律师把经验变成团队标准

成熟律师脑中有大量隐性经验：什么条款通常不可接受、哪些行业有特殊红线、哪些客户愿意让步、哪些问题必须升级。这些经验如果只存在个人脑子里，团队很难复制。

锚点可以把这些经验逐步变成规则草案、规则集和 action 模板。未来的前端应当让资深律师能用自然语言写下审查标准，再由系统生成待审核规则草案，最后由律师确认后进入团队 playbook。

### 4.3 帮律师管理注意力

成熟律师不是要系统把所有内容都标满。他更需要系统帮他区分轻重：哪些是 critical，哪些只是提示；哪些是缺失条款，哪些是谈判点；哪些需要升级，哪些可以交给 junior 先处理。

因此，锚点不能被做成噪音列表。界面需要有过滤、分组、优先级、状态和 action queue，让律师把有限注意力放在最重要的法律事项上。

### 4.4 帮律师把判断转化为动作

成熟律师最终不是为了“发现风险”而工作，而是为了推动事情完成：改条款、问业务、升合伙人、要求对方让步、接受某个风险、形成记录。

这就是 Action Anchor 的意义。前端要把“发现”自然推进到“处理”：每个重要锚点都能进入 action queue，并显示当前状态、负责人、处理意见和审计记录。

### 4.5 帮律师解释自己为什么这样做

律师的专业责任要求他能够解释自己的判断。客户、合伙人、对方律师、审计人员都可能追问：为什么这个条款要改？为什么这个风险可以接受？为什么当时没有升级？

锚点的价值不是替律师回答这些问题，而是把证据链保存好：原文、规则、风险、建议、处理动作、更新时间和处理人。这些信息应当自然出现在工作台和审计时间线里。

从这个角度看，锚点的用户不是“想让系统替自己审合同的人”，而是“希望把自己的专业判断做得更稳、更快、更可复用、更可追溯的律师”。这也是未来律所和法务团队愿意采用锚点的根本原因。

## 5. 为什么锚点不替律师做判断

这是产品最重要的边界。

产品上不要这样说：“本合同不能签。”

它应该说：“以下位置存在无限责任风险；当前规则建议增加责任上限；该问题属于 critical；建议法律审查员确认是否要求修改。”

原因很实际。

### 5.1 法律判断依赖客户目标

同一个条款，在不同交易里结果可能完全不同。

例如无限责任：

- 对供应商来说，通常是重大风险。
- 对客户来说，可能是谈判目标。
- 对低金额、短期、战略客户项目，处理方式又可能不同。

系统不可能完整知道客户真实商业目标，所以不能代替律师下最终判断。

### 5.2 法律判断依赖上下文

合同风险不只来自一句话。它可能和合同金额、行业监管、交易结构、历史谈判、客户偏好、对方强势程度有关。

后台可以把证据找出来，但是否接受、如何修改、是否让步，仍然需要律师判断。

### 5.3 律师需要可解释证据，而不是黑盒结论

律师不能向客户说“系统认为有风险”。律师需要说：

```text
第 12 条约定责任不设上限，与公司 playbook 中责任限制政策冲突，因此建议加入 cap。
```

界面要帮助律师把这条证据链讲清楚。

### 5.4 产品信任来自克制

法律科技产品最容易失败的地方，是表现得像“替代律师”。真正能进入律所和企业法务日常的产品，通常是“增强律师”：更快发现问题、更稳留痕、更好协作、更容易复用经验。

因此，产品文案要避免：

- “系统已判定”。
- “本合同合规/不合规”。
- “必须修改”。
- “自动法律意见”。

推荐使用：

- “发现”。
- “提示”。
- “建议复核”。
- “建议升级”。
- “建议加入 action”。
- “律师确认后生效”。

## 6. 锚点如何帮助律师日常工作

### 6.1 合同初筛

律师每天会面对大量合同。初筛的目标不是写最终意见，而是快速知道：

- 哪些合同风险高。
- 哪些条款需要重点看。
- 哪些合同缺核心条款。
- 哪些问题可以交给 junior 或业务先处理。
- 哪些问题必须升级。

因此，首屏需要是“审查工作台”：上传或粘贴合同后，左侧看文档，右侧看 anchor，顶部看风险概览。

### 6.2 缺失条款检查

很多合同问题不是写错了，而是没写。

例如：

- 没有责任上限。
- 没有数据删除/返还义务。
- 没有保密期限。
- 没有终止后义务。
- 没有争议解决条款。

Missing Anchor 对前端很特殊：它没有文字位置。不要在正文里硬找位置。更合适的做法，是放在侧栏或“缺失问题”分组中，并提示律师：这是文档级缺失，不是某一句话的命中。

### 6.3 风险审查

Risk Anchor 帮律师把问题分轻重缓急。界面至少要支持：

- 按严重程度筛选：critical/high/medium/low。
- 按风险类型筛选：liability、data、payment、termination 等。
- 显示推荐处理方式。
- 显示规则来源：base/industry/organization。
- 让律师标记“接受风险”“要求修改”“需要升级”。

### 6.4 义务抽取

Obligation Anchor 适合做履约和交割管理。

律师关心的是：

- 谁要做？
- 做什么？
- 何时做？
- 什么条件触发？
- 不做会有什么后果？

义务适合做成表格：主体、义务、期限、触发条件、来源条款、状态。

这会让 LexAnchor 不只是合同审查工具，也能变成合同履行和项目交付的工作台。

### 6.5 条款关系理解

Relation Anchor 用来解释条款之间的连接。

例如：

```text
终止条款 -> 触发数据返还义务
SLA 条款 -> 触发 service credit remedy
保密义务 -> 延续至合同终止后
```

关系可以用卡片展示，不一定要做复杂图谱。律师最需要的是看懂：一个条款会影响另一个条款，修改时不能只改单点。

### 6.6 Action Queue

Action Anchor 是把审查发现转成工作事项。

例如：

- 要求加入责任上限。
- 升级数据处理条款给隐私律师。
- 要求业务确认自动续约是否可接受。
- 要求对方删除不合理赔偿范围。

产品需要有 action queue，而不是只显示 findings。律师最终工作不是“看风险列表”，而是“处理风险”。

目前后台支持这些 action 状态：

```text
proposed -> accepted -> in_progress -> completed
proposed -> dismissed
```

按钮应当对应这些真实动作。

## 7. 和现有产品的比较

### 7.1 和传统 CLM 的区别

CLM 通常关注合同生命周期：起草、审批、签署、归档、续约。它擅长流程，但不一定深入解释“为什么这个条款有法律问题”。

LexAnchor 更像 CLM 中的法律审查引擎：

- CLM 管流程。
- LexAnchor 固定审查证据、风险、义务和动作。

未来可以和 CLM 集成，但 LexAnchor 自己不应退化成普通流程表单。它的核心体验仍然是“律师审查工作台”。

### 7.2 和 Word 批注/红线的区别

Word 批注是人工记录，优点是律师熟悉，缺点是难以结构化、难以统计、难以复用。

LexAnchor 的优势：

- 每个问题有结构化类型。
- 可以按风险、义务、升级、动作筛选。
- 可以形成审计记录。
- 可以沉淀规则，下一次自动复用。
- 可以生成报告和 action queue。

产品不必完全复制 Word。更好的做法，是保留律师熟悉的文档视图，同时增加结构化侧栏和任务流。

### 7.3 和普通智能合同审查工具的区别

很多合同审查工具会输出一段总结或 checklist。问题是，律师很难知道它依据哪里、漏了什么、能否复核。

LexAnchor 的区别：

- 输出不是一段泛泛建议，而是 anchored findings。
- 每个 finding 有规则、证据、严重程度、推荐处理。
- 缺失条款不会伪造位置。
- 高风险问题可以进入 escalation 和 action。
- 所有读取和更新都有审计记录。

这更符合律所和企业法务对“可解释、可复核、可留痕”的要求。

### 7.4 和知识库/Playbook 的区别

Playbook 告诉律师应该怎么审，但通常是静态文档。

LexAnchor 把 playbook 变成可运行规则：

```text
律师指南 -> 规则草案 -> 律师审核 -> 规则集 -> 审查发现 -> action
```

目前后台已经支持从律师自然语言指南生成 disabled rule drafts。界面要强调：规则草案必须经过律师审核，不能自动启用。

### 7.5 和 eDiscovery/检索工具的区别

检索工具帮律师找文档或关键词。LexAnchor 关注的是合同审查中的法律判断坐标。

检索回答：“哪里出现了 unlimited liability？”

锚点回答：“这里出现 unlimited liability，它触发了责任风险规则，严重程度 critical，建议加入 cap，并生成一个 action 让律师处理。”

## 8. 为什么律所和法务团队会选择锚点

### 8.1 它不挑战律师的专业地位

许多法律科技产品让律师不舒服，是因为它们看起来像在替律师做结论。LexAnchor 的姿态更适合专业服务：系统提供证据和流程，律师保留判断权。

### 8.2 它能提高 junior 工作质量

Junior 或实习律师常见问题不是不努力，而是不知道哪些点必须看。锚点可以把合伙人或资深律师的 playbook 前置，让 junior 至少不会漏掉基础红旗。

### 8.3 它能沉淀律所知识资产

律所最有价值的是经验和判断标准，但这些东西通常分散在律师脑子里、历史批注里、群消息里、旧文件里。

LexAnchor 可以把这些经验转成规则草案和规则集，逐步形成团队自己的审查体系。

### 8.4 它能让审查过程可追溯

客户、合伙人、风控、审计都关心：谁看过、什么时候看过、为什么这么处理。

目前后台已经有 audit events。审计不应只是后台日志，而应成为律师可以理解的时间线。

### 8.5 它适合不同规模团队

小团队可以用它做快速审查和报告。

大团队可以用它做规则治理、分工协作、action queue、审计中心。

### 8.6 它有利于标准化服务交付

律所常常希望同类合同由不同律师审查时，结果不要差太多。锚点可以把底线规则、行业规则、客户规则分层，减少纯个人经验差异。

## 9. 前端的产品形态

第一屏不要做成营销首页，也不要做成“上传文件后给一段总结”的轻工具。

LexAnchor 的前端，本质上是一个法律审查工作台。

### 9.1 首屏：Review Workspace

核心布局建议：

```text
顶部：当前组织 / 工作区 / 用户 / 规则集 / 扫描状态
左侧：合同文档视图或文本视图
右侧：Anchor 面板
底部或右下：Action Queue / Audit Timeline 入口
```

Anchor 面板建议有这些 tab：

- Overview：风险总览、严重程度、扫描版本、规则集。
- Findings：全部 findings，可筛选。
- Missing：缺失条款。
- Obligations：义务表。
- Relations：条款关系。
- Actions：待处理 action。
- Audit：审计时间线。

### 9.2 合同上传/粘贴入口

律师应能两种方式开始：

- 粘贴合同文本，调用 `POST /api/v1/contract/scan-text`。
- 上传文件，调用 `POST /api/v1/contract/scan` 或 `POST /api/v1/contract/scan-async`。

不要一开始让律师填写太多技术参数。默认使用 `rules_v1.0`，高级选项折叠起来。

建议默认：

```text
ruleset = rules_v1.0
semantic_validation = validate
action_policy = default
grounding_backend = auto
```

### 9.3 Anchor 详情卡片

每个 anchor 卡片建议显示：

- 标题：规则名或问题名。
- 类型：Text/Missing/Risk/Obligation/Action 等。
- 严重程度：critical/high/medium/low。
- 状态：confirmed/candidate/proposed/completed 等。
- 证据：matched text 或缺失说明。
- 规则来源：base/industry/company。
- 推荐处理：recommendation。
- 操作：接受、忽略、升级、创建/更新 action。

卡片文案要像律师助理，不要像机器判决。

推荐写法：

```text
发现潜在无限责任风险，建议复核是否需要加入责任上限。
```

不推荐：

```text
本条款违法，必须修改。
```

### 9.4 Action Queue 页面

Action Queue 是前端完整度的关键。没有 action queue，系统只是报告工具；有 action queue，系统才进入律师日常工作。

建议列表字段：

- Priority。
- Action title。
- Owner role。
- Source finding。
- Due policy。
- Status。
- Last updated。
- Comment。

操作按钮：

- Accept。
- Mark in progress。
- Complete。
- Dismiss。
- Add comment。

对应 API：

```text
GET   /api/v1/jobs/{job_id}/actions
PATCH /api/v1/jobs/{job_id}/actions/{action_id}
```

### 9.5 Rule Authoring 页面

这个页面给资深律师或知识管理律师使用。

它的目的不是让普通用户随便改规则，而是把律师指南转成可审核的规则草案。

页面流程：

1. 输入 playbook 或审查指南。
2. 点击生成草案。
3. 展示生成的规则、关键词、严重程度、负面过滤、建议动作。
4. 展示 review checklist。
5. 明确提示：草案默认不启用，必须律师审核。

对应 API：

```text
POST /api/v1/rule-authoring/draft-from-text
```

### 9.6 Audit Center 页面

审计不是给程序员看的日志，而是给法律团队看的责任记录。

页面应显示：

- 谁上传了合同。
- 谁查看了报告。
- 谁更新了 action。
- 谁生成了规则草案。
- 时间、资源类型、结果。

对应 API：

```text
GET /api/v1/jobs/{job_id}/audit-events
GET /api/v1/audit/events
```

### 9.7 Admin / Membership 页面

目前后台已经支持 membership-backed RBAC。前端可以先做一个简单成员管理页面：

- 用户 id。
- organization。
- workspace。
- roles。
- status。

对应 API：

```text
POST /api/v1/memberships/dev-grant
GET  /api/v1/memberships
```

当前这是开发级管理能力。未来要替换为真实身份系统和 matter-level assignment。

## 12. 前端文案原则

法律产品的文案会直接影响用户信任。

推荐：

- “发现 3 个高风险锚点”。
- “建议律师复核”。
- “可能缺少数据返还义务”。
- “该问题已加入 action queue”。
- “规则草案尚未启用，需审核”。

避免：

- “系统已经判断合同不可签”。
- “系统保证无风险”。
- “自动法律意见”。
- “本条款违法”。
- “无需律师复核”。

前端应不断提醒用户：LexAnchor 是审查辅助和证据固定工具，不是最终法律意见系统。

## 14. 前端设计时要避免的误区

### 14.1 不要只做聊天框

聊天适合问答，但合同审查需要证据、位置、状态、动作、审计。LexAnchor 前端应以工作台为主，聊天最多是辅助入口。

### 14.2 不要只显示总分

“风险 85 分”对律师帮助有限。律师需要知道具体哪里有问题，为什么有问题，下一步怎么处理。

### 14.3 不要把 Missing Anchor 当成正文高亮

缺失问题没有原文位置。它应该显示为文档级审查事项。

### 14.4 不要隐藏规则来源

律师会问：“为什么系统说这是风险？” 前端必须显示规则名、规则来源、触发证据。

### 14.5 不要自动启用生成规则

规则是律师知识资产。自动生成可以节省整理时间，但启用必须经过律师确认。

## 15. 如何验收第一版前端

第一版前端至少要能完成这条路径：

```text
登录 -> 上传/粘贴合同 -> 查看锚点 -> 查看报告 -> 处理 action -> 查看审计
```

验收问题：

- 律师能否在 30 秒内知道合同最重要风险？
- 每个风险能否看到对应证据或缺失说明？
- 律师能否区分“系统发现”和“律师确认”？
- action 是否能被推进，而不是只停留在报告里？
- 规则草案是否明确标记为未启用、待审核？
- 审计记录是否能说明谁做了什么？
- 非技术用户是否能不看 API 文档完成主要工作？

## 17. 未来锚点规则库如何开发

锚点真正长期有价值的部分，不只是一次扫描合同，而是逐步形成一个属于律所或法务团队自己的规则库。

这个规则库不是普通知识库，也不是几份 Word playbook 的搬运。它更接近“可运行的法律经验”：资深律师过去依靠经验识别的问题，未来可以逐步变成系统能提示、能解释、能进入 action queue、能被团队复用的规则。

### 17.1 规则库开发的基本过程

未来规则库开发可以按这条路径推进：

```text
律师经验 -> 审查指南 -> 规则草案 -> 律师审核 -> 测试样本 -> 规则版本 -> 团队启用 -> 使用反馈 -> 持续迭代
```

第一步是收集律师经验。来源可以是历史合同批注、客户 playbook、谈判清单、合伙人审查意见、行业监管要求、争议案例、内部培训材料。

第二步是把经验整理成审查指南。例如：“供应商合同中不得接受无限责任”“客户数据必须有返还或删除义务”“自动续约必须提前通知业务确认”。

第三步由系统生成规则草案。目前后台已经支持从律师自然语言指南生成 disabled rule drafts。界面需要明确展示：这些只是草案，不会自动启用。

第四步由律师审核。律师需要确认规则名称、触发关键词、适用范围、风险等级、负面过滤、推荐动作、是否需要升级。

第五步用测试样本验证。每条规则至少要有正例和反例，避免误报太多，也避免漏掉关键风险。

第六步版本化启用。规则应区分 base、industry、organization、workspace 或 client playbook。启用时要能看到版本、来源、审核人和生效范围。

第七步根据使用反馈持续迭代。律师在日常审查中接受、忽略、升级、修改 action 的行为，都可以反过来帮助团队优化规则。

### 17.2 规则库相对其他系统的优势

相对传统知识库，锚点规则库不是静态阅读材料，而是能在合同审查时自动工作。律师不需要想起某份 playbook 放在哪里，系统会在相关条款出现时提示。

相对提示词式工具，锚点规则库更稳定。提示词容易因为表达方式不同而输出不一致，规则库则有明确规则 id、适用范围、严重程度、证据字段和审计记录。

相对 CLM 模板库，锚点规则库更关注审查判断。模板库解决“从哪里起草”，锚点规则库解决“审查时什么值得注意，为什么值得注意，下一步怎么处理”。

相对 Word 批注经验，锚点规则库可以复用、统计、审计和迭代。一个资深律师的经验不再只留在某份旧合同里，而可以变成整个团队的审查标准。

相对一次性合同总结，锚点规则库更适合律所和企业长期使用。它能沉淀客户偏好、行业风险、团队标准和历史处理方式，形成持续增强的专业资产。

### 17.3 前端如何体现规则库价值

Rules 页面不宜做成技术配置页。它更像一个“法律知识资产管理台”。

建议至少展示：

- 规则名称和适用范围。
- 规则来源：base、industry、company、workspace、client。
- 风险等级和 anchor 类型。
- 触发证据和负面过滤。
- 推荐动作和升级策略。
- 草案状态：待审核、已通过、已启用、已停用。
- 版本、审核人、更新时间。
- 最近命中次数、被接受次数、被忽略次数。

这样前端才能让律师理解：规则不是程序员写的配置，而是团队法律经验的资产化表达。

## 18. 未来锚点如何宣传

宣传时不要把重点放在“自动替你审合同”。这会让专业律师天然警惕，也容易把产品放进同质化工具竞争里。

更好的方向是强调：锚点让律师的判断更稳、更快、更可复核、更可传承。

### 18.1 核心市场定位

建议定位为：

```text
面向律所和企业法务的法律审查基础设施。
```

它不是聊天机器人，不是普通合同摘要，也不是 CLM 的流程插件。它是把合同审查中的法律证据、风险、义务、动作和团队规则结构化的后台能力。

### 18.2 对律所的宣传重点

对律所，应强调：

- 帮 junior 少漏关键风险。
- 帮合伙人把审查标准沉淀成规则库。
- 帮团队提高同类合同审查的一致性。
- 帮客户看到更清楚的风险证据和处理记录。
- 帮律所形成可复制、可交付、可解释的法律服务产品。

适合的市场语言：

```text
把合伙人的审查经验，变成团队每天可运行的规则。
```

### 18.3 对企业法务的宣传重点

对企业法务，应强调：

- 快速识别高风险合同。
- 把公司 playbook 落到每一次审查。
- 让业务提交、法务审查、风险升级有清楚状态。
- 让审计和管理层能追踪谁处理了什么。
- 降低合同审查结果因个人经验不同造成的波动。

适合的市场语言：

```text
让每一份合同都按公司的法律底线被检查、被记录、被推进。
```

### 18.4 对法律科技市场的差异化表达

市场上已经有很多“智能合同审查”“合同总结”“智能问答”工具。LexAnchor 要避开同质化表达，突出三点不同：

- Anchored：每个发现都有证据位置或明确缺失说明。
- Reviewable：律师可以复核、接受、忽略、升级、形成 action。
- Governable：规则、权限、审计、版本可以管理。

宣传时可以反复强调：

```text
不是让机器给结论，而是让律师的判断有坐标。
```

### 18.5 内容营销方向

未来可以围绕这些主题做市场内容：

- “为什么合同审查不应该只看自动总结”。
- “成熟律师审合同真正看什么”。
- “从 playbook 到可运行规则库”。
- “律所如何把合伙人经验复制给 junior”。
- “企业法务如何减少合同审查中的漏点和反复沟通”。
- “为什么缺失条款比问题条款更难发现”。
- “合同审查的下一步不是聊天，而是 action queue”。

这些内容会吸引真正有法律工作痛点的用户，而不是只被概念热度吸引的用户。

## 19. 推广口号

一个好的口号应该同时表达理念和市场价值：锚点不替代律师，而是让律师的判断更清楚、更可靠、更可传承。

主口号建议：

```text
LexAnchor：让每一个法律判断，都有可复核的坐标。
```

这个口号适合放在官网首页、产品介绍、销售材料和演示开场中。它强调三件事：

- “法律判断”说明产品服务的是律师专业工作，而不是普通文本处理。
- “可复核”说明产品尊重律师判断和专业责任。
- “坐标”说明锚点的核心能力是把判断固定到证据、规则、动作和审计链上。

备选口号：

- `把律师经验，变成可运行的规则。`
- `不替律师判断，只让判断更有依据。`
- `从合同文字，到法律动作。`
- `让合同审查可解释、可协作、可传承。`
- `每一个风险，都找到它的证据和下一步。`

如果只能选一句用于长期推广，建议使用：

```text
让每一个法律判断，都有可复核的坐标。
```

## 20. 给前端负责人的最终建议

你不是在做一个“自动合同总结器”。你是在做一个律师工作台。

这个工作台要让律师感到：

- 我仍然掌握最终判断。
- 系统帮我少漏点。
- 系统把证据和理由整理清楚。
- 系统能把问题变成行动。
- 系统能沉淀我的经验。
- 系统能在团队里形成一致标准。

只要前端把这六件事做好，LexAnchor 就不只是一个后台 API，而会成为律所和法务团队愿意长期使用的审查基础设施。
